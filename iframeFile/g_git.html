三、Git命令全局配置

  • 命令描述命令描述
    $ git config --list查询git 配置$ git config -e修改当前仓库下git的配置文件 -e，e即edit，编辑的意思
    $ git config -e --global修改机器上所有仓库的配置文件$ git config --global -l查询git 设置
    $ git config --global core.editor "vim"键入以下命令以设置您的首选文本编辑器：“nano”、“vim”、“emacs”、“notepad”$ git config --global color.ui auto键入以下命令以使 Git 能够使用颜色：
    $ git config --global user.name查询git 名称配置$ git config --global user.email查询git 邮箱配置
    $ git config user.name$ git config user.email
    $ git config --global user.name要切换 Git 帐户，您需要更新与您的 Git 配置关联的用户名和电子邮件$ git config --global user.email要切换 Git 帐户，您需要更新与您的 Git 配置关联的用户名和电子邮件
  • 注意事项：
    1. 1、设置签名（实际开发的时候通常只设置一个系统用户级别）
    2. 2、这里设置的签名和登录远程库的帐号密码没有任何关系
    3. 3、级别优先级：就近原则，项目级别优先于系统用户级别，二者都有时，采用项目级别的签名
    4. 4、如果只有系统用户级别的签名，就以系统用户级别的签名为准，二者都没有，不允许操作下一步
    5. 5、项目级别/仓库级别：仅在当前本地库范围内有效 git config user.name // 信息会保存到 ./.git/config
  • 系统用户级别：登录当前操作系统的用户范围，全局 git config --global user.name // 系统文件会在./user/xiongbab/.gitconfig

九、状态

  • 命令描述
    $ git status查看Git 状态
    $ git status -s以短格式输出状态，或者 --short
    $ git status -b显示当前分支的名称，并显示工作目录是否比远程分支新或旧。或--branch
    $ git status -u显示未跟踪的文件。可使用可选的{mode}参数指定显示未跟踪文件的级别。
    可选值为no、normal、all。默认情况下，不带参数的-u选项显示普通级别的未跟踪文件。
    例如：--untracked-files[=<mode>]，输出 "?? newfile.txt" 表示有一个未跟踪的文件newfile.txt。
    $ git status --ignored显示被忽略的文件。此选项将列出在.gitignore文件中指定的忽略文件
    other* 代表忽略xx开头的文件 输出 "?? newfile.txt" 表示有一个未跟踪的文件newfile.txt。
    $ git status --column在短格式输出中使用列布局。默认情况下，短格式输出使用行布局。
    输出 " M example.txt" 表示文件example.txt已修改但未暂存，其中状态码在两列中间对齐。
    $ git status --long以长格式输出状态。输出中包含更多的信息，如文件类型、文件的最新修改时间和文件的权限
    输出 "modified: example.txt\n" 表示文件example.txt已修改，同时显示文件的详细信息。

十三、合并冲突解决,比较文件差异

  • 命令描述
    $ git merge user合并分支，如果有冲突文件，后面会带 （master|MERING），找到对应合并文件手工解决
    $ git add xxx手工在文件里处理完后，提交下就可以合并了
    $ git diff在解决冲突之前，可以先diff 查看文件中的冲突，不带文件名，可以比较多个

    cat {file.txt}当执行了git merge 合并分支时，当发生两个文件合并冲突时，手动保留代码

十五、实现版本的前进回退

  • 命令描述
    $ git reflog获取历史版本号，日志多一点参数，可查看版本，确定版本的索引号
    $ git log --oneline简写形式，哈希值会缩减
    $ git log --oneline --graph --decorate分支的合并后显示log
    $ tail -n 3 good.text基于索引值操作版本前进后退 [推荐]
    $ git reflog先看版本，确定版本索引号
    $ git reset --hard {哈希值}使用reset 重置的意思，--hard 880692b，后退/前进到指定版本来找回被删除的文件
    $ git reset --hard {版本号}回退到该版本号对应的版本
    $ git reset --hard HEAD/commit_id回滚最近的一个版本 git log
    $ git reset --hard HEAD^基于^后退(只能后退),后退一个版本,使用后,log只显示历史的版本，不显示最新
    $ git reset --hard HEAD^^^后退几个版本 取决于 ^ 号版本
    $ git reset --hard HEAD~3基于 ~ [步数] 后退几步,注意 HEAD 要大写 ，~[退的版本步数],reset命令的三个参数对比
    --soft 在本地库移动HEAD指针，仅仅
    --mixed 在本地库移动HEAD指针，但会重置暂存区
    --hard 在本地库移动HEAD指针，且重置暂存区，并重置工作区
    $ git-revert是进行新的提交，以还原其他提交所做的更改。
    $ git revert {版本号}新的提交，不会删除原提交，用于恢复之前的文件，且保留之前的提交信息
    $ git-restore使得在工作空间但是不在暂存区的文件撤销更改(内容恢复到没修改之前的状态)
    $ git restore --staged是将暂存区的文件从暂存区撤出，但不会更改文件的内容
    $ git retore a.text撤销被删除的文件，并恢复到工作区，但是如果删除了并提交了，那就恢复不了
    $ git reset --hard {hard}重置，恢复到某个提交的位置，并恢复文件，但会删除之前提交的版本号

二十一、跨团队协作

  • 先把自己的远程库地址发给其他团队的人员，让他们fork
  • 其他团队的人员登录自己的帐号，然后点开发来的远程库地址链接，点击右上角fork
  • Fork 后，在远程库下的项目名称下，会出现原作者的来源地址
  • 终端下，创建一个自己要克隆存放的目录
  • git clone [远程库地址HTTPS] 如果出现克隆失败，需要把https:// 改成 git://
  • 其他团队的人员在自己开发完自己的功能后做本地修改，提交git commit操作
  • git push origin master （这里是登录其他团队的人员帐号来发起合并操作）推送远程
  • 其他团队的人员帐号的远程库下，点 Pull requests 点完再点 Creat pull request ，提交信息
  • 对方已经提交好了，这里需要登录自己的帐号，然后进入项目 Pull requests
  • 点进内容后，可以先进行对话，可查看操作了什么 ，具体文件修改了什么，就是审核
  • 然后确认完后，点击Merge pull request 合并 ，添加合并的日志信息，Confirm merge
  • 然后拉取到本地，git pull origin master
  • 结束

二十三、分支的概念

  • 同时并行推进多个功能的开发，提高开发效率

二十四、哈希的概念

  • 明文 => 加密算法 => 密文
  • 工具：Hash - 1.04
  • 哈希是一个系列的加密算法，各个不同的哈希算法虽然加密强度不同，但是有以下几个共同点：
    1. 1. 不管输入数据的数据量有多大，输入同一个哈希算法，得到的加密结果长度固定。
    2. 2. 哈希算法确定，输入数据确定，输出数据能够保证不变
    3. 3. 哈希算法确定，输入数据有变化，输出数据一定有变化，而且通常变化很大
    4. 4. 哈希算法不可逆
    5. 5. Git底层采用的是SHA-1算法。

二十五、Git 保存版本的机制

  • 集中式版本控制工具的文件管理机制 ,以文件变更列表的方式存储信息。这类系统将它们保存的信息看作是一组基本文件和每个文件随时间逐步累积的差异。
  • git 的文件管理机制 ,Git把数据看作是小型文件系统的一组快照。每次提交更新时Git都会对当前的全部文件制作一个快照并保存这个 , 快照的索引。为了高效，如果文件没有修改，Git不再重新存储该文件，而是只保留一个链接指向之前存储的文件。 ,所以Git的工作方式可以称之为快照流。
